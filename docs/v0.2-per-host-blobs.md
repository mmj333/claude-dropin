# v0.2 — per-workstation encrypted blobs (deferred)

## Motivation

v0.1 ships one encrypted blob (`config/claude.age`) containing every
session across every customer box ever visited. Two problems:

1. **Blast radius** — during an active session on customer X, *every*
   customer A–W session is also in plaintext on disk (`$SCRATCH`).
   Disk compromise mid-session exposes the full corpus.
2. **Decrypt cost** — the blob grows linearly with total sessions, and
   the full blob is decrypted+extracted on every launch. Gets slow on
   USB / spinning disks after tens of customer visits.

## Sketch

```
config/
  identity.age              # shared — one passphrase, one X25519 keypair
  recipient.pub             # shared
  blobs/
    <hostname>.age          # per-workstation encrypted session state
    <hostname>.age          # ...
```

Identity and recipient stay global: one passphrase, one keypair. Only
the session state (the heavy part) splits per-host.

## Flow changes

- **`run.*`**: determine hostname. If `blobs/<hostname>.age` exists,
  decrypt it; else seed a fresh blob from `templates/<platform>/` (same
  logic `setup.*` uses today).
- **`eject.*` / EXIT trap**: always encrypt scratch to
  `blobs/<hostname>.age` (not the legacy `claude.age`).
- **`setup.*`**: no longer writes a session blob, just the global
  identity + recipient. First-time host visit materializes the blob.
- **v0.1 → v0.2 migration**: on first v0.2 launch, if a legacy
  `config/claude.age` is present and `config/blobs/` is empty, rename
  to `config/blobs/migrated.age`. Log a note so the user can manually
  re-split later if they care.

## Implications

- Only the current customer's state is in plaintext at runtime. Blast
  radius of a mid-session disk attacker = one customer.
- Decrypt time becomes O(sessions-on-this-customer), not O(total).
- `CLAUDE_DROPIN_SHARED_WORK=1` becomes roughly redundant — per-host
  blobs imply per-host work by construction. Likely retire it and
  replace with a single `CLAUDE_DROPIN_SHARED_BLOB=1` opt-out for the
  "I want one pile" case.
- Minor: the on-disk size (plain + encrypted overhead) grows slightly
  because each blob carries its own age header. Negligible vs. the
  plaintext growth we avoid on decrypt.

## Future layering (v0.3+)

- Sync `blobs/<hostname>.age` to a server over WireGuard on eject;
  fetch on run. Decoupled from the per-host layout — just a "where
  does the blob live" question. Per-host layout is the enabler.
- Per-host *identity* (not just per-host blob): each customer gets its
  own X25519 keypair wrapped with the same passphrase. Adds
  compartmentalization even against an attacker with the passphrase.
  More moving parts; probably not worth it vs. per-host blob alone.

## When to implement

Defer until v0.1.x has accumulated real session history in field use.
The migration is trivial (one rename) so nothing's lost by waiting.
